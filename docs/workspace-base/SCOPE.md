# @adhd/workspace — Scope (reframed 2026-06-26: agent-optimized, self-enforcing workspace)

> A workspace-standards system that **optimizes a monorepo for AI agents, not human browsing**: route a
> task to the right scope, load only the rules that apply, see the blast radius, and execute under
> machine-enforced invariants — in minimal tokens. The standard is *generated by inheritance*,
> *upgradeable from one place*, and *gated by lint* so any drift fails CI.
>
> **Split into a platform-agnostic core + thin ecosystem adapters** so the reusable parts are not confined
> to nx/Node. Tracks BACKLOG `FEAT-WORKSPACE-001` (+ the broken-out features in §10). Supersedes the
> 2026-06-26 single-package `@adhd/workspace-base` scope; the locked Q1–Q8 decisions are preserved below,
> reorganized under the new architecture.
>
> **Aligns with** the sox `memory-refactor` nx-generator handoff
> (`sox-ecosystem/docs/plan/memory-refactor/NX-GENERATOR-HANDOFF.md`) — the same born-conformance generator
> at a more mature layout/boundary stage. §9 reconciles the two; the generic core is intended to be
> consumable by **both repos**.

## 0. Reframe — the repo as an agent decision-routing system  ·  (memory 01KW2GHJE)
Optimizing layout for agents reframes the folder hierarchy as a **context-partition mechanism** (Claude Code
auto-loads `CLAUDE.md` from cwd upward), so `area/group/package` is a *context-scoping unit*, not just
organization. The deterministic agent task loop the standard enables:

```
discover (generated index) → scope (auto-loaded CLAUDE.md) → debug (impact graph)
  → recall (soft memory) → execute (lint-enforced invariants) → learn (reflection tagged by group)
```

…replacing broad grep/read exploration. Five layers, with a strict division of labor:

| # | Layer | Answers | Source | Discipline |
|---|---|---|---|---|
| 1 | **Generated routing index** (`map.json` + `INDEX.md`, root + per-area) | DISCOVERY (concept/tag → area/group/package/path/deps) | build/project graph + per-package metadata block (§3) | **GENERATED**, drift-gated — never hand-maintained |
| 2 | **Authored hierarchical CLAUDE.md** (root → area → group → package) | EXECUTION (invariants, build/test cmds, footguns) | human/agent intent that can't be derived | **AUTHORED**; auto-scoped by cwd; links to the index, never duplicates it |
| 3 | **Impact graph** (code-intelligence + affected targets) | DEBUGGING (callers, flows, blast radius, what to rebuild/test) | the build graph | derived on demand |
| 4 | **Project memory** (semantic store, tagged by group) | learned lessons recalled at task time | reflections | **SOFT** — advisory, never gating; correctness must not depend on recall health |
| 5 | **Intent→scope router** ("if doing X, go to Y") for the top ~20 task types incl. cross-cutting | ROUTING (where the most discovery tokens are wasted) | hand-curated | small, authored, high-leverage |

**Core invariants of the pattern (the load-bearing rules):** GENERATE the routing table (a hand-maintained
mirror drifts) · AUTHOR the rulebook (intent only, no derivable facts) · keep MEMORY soft · **promote
invariants from prose to machine-checkable gates** (module-boundary tags / lint / drift checks) so "execute
correctly" is *enforced*, not hoped.

### 0a. The five layers ARE generator + linter logic (not just docs)
Each reframe layer has a **generate side** (the generator produces/refreshes it) and a **check side** (the
linter fails CI on drift). This is how the reframe is *implemented*, not merely described:

| Layer | Generator produces / refreshes | Linter gate (fails CI on…) |
|---|---|---|
| 1 · routing index | (re)generates `map.json` + `INDEX.md` (root + per-area) from the metadata blocks + project graph | index drift vs the live graph; an area/group not declared in the taxonomy (§8) |
| 2 · CLAUDE.md hierarchy | scaffolds the root→area→group→package `CLAUDE.md` stubs with managed regions | a missing CLAUDE.md in the hierarchy; a managed region whose marker tags are absent (unmanaged) |
| 3 · impact graph | nothing to stamp (derived on demand) | n/a — consumed at task time via the adapter's `affected` binding |
| 4 · soft memory | nothing (advisory) | **never gates** (correctness must not depend on recall) |
| 5 · intent→scope router | scaffolds/extends the top-~20 `ROUTER.md` entries, validating each target scope resolves | a router entry pointing at a scope/path that doesn't exist; a new top-level concern with no router entry |

Plus the always-on gates from §4–§9: required targets/files/sections, boundary policy, post-change rules,
provenance. The generator's job is to make a package **born-conformant**; the linter's job is to keep it that
way. Both read the same source of truth — the **taxonomy** (`.adhd/workspace.json`, §8) and the per-package
**metadata blocks** (`.adhd/meta.json`, §3).

## 1. Architecture — generalizable core vs ecosystem adapters  ·  (the "break it out of nx" cut)

The reusable standard is **data + checkers operating on files, a metadata manifest, and git** — none of which
needs nx. The nx/Node specifics are an adapter. Three layers:

```
@adhd/workspace-standard   ← platform-AGNOSTIC core. Pure TS, zero nx/devkit import. Node CLI, but what it
  (the "generalizable")      CHECKS is language-agnostic (validates a Python/Rust/Go package equally).
        ▲
        │ consumed by
        │
@adhd/workspace-nx         ← nx ADAPTER (the "nx-confined specifics"). Generators (post-generate hook),
  (the "nx-confined")        eslint-plugin wiring, `nx affected` impact-graph binding, esbuild/tsconfig
                             stamping, registry drift gate.
        ▲
        │ (future, same core)
@adhd/workspace-py | -cargo | -go   ← other ecosystem adapters, if/when needed. Out of scope now; the point
                                      is the core does not preclude them.
```

### 1a. `@adhd/workspace-standard` — the generalizable core (no nx)
Pure-TypeScript, `platform:shared`. Exposes the standard **as data + pure checkers + a portable FileSet
scaffold core** (the sox `libs/authoring` equivalent). Contains, all nx-free:

- **Required-targets registry** (§4) — the *list + semantics*, not the runner wiring.
- **Required-files + section rules engine** (§5) — existence + placeholder/section checks over a FileSet.
- **Managed-region engine** (§8) — marker insert/extract/idempotent re-apply; verifies marker presence.
- **Post-change enforcement rule engine** (§6) — "what must update when X changes", driven by a git diff +
  the metadata manifest. No nx.
- **Change-provenance schema + validator** (§7) — parse/validate the provenance trailer.
- **Routing-index format + drift gate** (§0 layer 1) — the `map.json`/`INDEX.md` *schema* and the
  generate-then-check drift gate. The *harvester* that reads the project graph is adapter-supplied via a
  narrow `GraphSource` interface; the format and drift logic are core.
- **Boundary-policy schema** (§9) — the `area→area` / `group→group` allow-matrix *as data*. The *enforcer*
  is adapter-supplied.
- **Per-package metadata block schema** (§3) — `{area, group, concerns, invariants, entrypoints}`.
- **Configurable taxonomy/layout map** (§8, Q8) — read from root config; neither `layer:*`/`platform:*`
  (adhd) nor `area:*`/`group:*` (sox) is hardcoded.

### 1b. `@adhd/workspace-nx` — the nx adapter
`platform:node`. Thin adapters over the core (mirrors the sox `sox-nx` ↔ `authoring` split, §6 of their
handoff):
- nx **generators** (`library`/`app`/`extension`) that call `applyWorkspaceStandard(tree, project)` as their
  final step, + a root post-generate hook (Q1).
- `@adhd/eslint-plugin-workspace` — wires the boundary-policy data into `@nx/enforce-module-boundaries` and
  fails CI on any missing target/file/field/marker/post-change-rule (Q3).
- **impact-graph** binding = `nx affected` / project-graph as the layer-3 source.
- esbuild self-contained build conventions, tsconfig/outDir stamping, the `registry/index.json` checksum
  drift gate (sox-specific carry; §9).

## 2. Problem
Every project is hand-rolled; targets/docs/tags/publish-config drift per-package; generators
(`@adhd/apigen-nx`, `generate-lib.sh`, sox `sox-nx`) each re-implement scaffolding with no way to (a)
guarantee a new project meets the standard, (b) roll a standard *change* out to all existing projects, or
(c) let an agent route a task without broad exploration. Unenforced live-test + demo standards let real bugs
hide. This system makes the standard a single inheritable, upgradeable, lint-enforced, **agent-routable**
source of truth — and makes its reusable parts available to any repo, not just this nx one.

## 3. Per-package metadata block (the routing source of truth)  ·  (harvested from sox §2)
Every package carries a structured metadata block at **`<pkg>/.adhd/meta.json`** — deliberately *not* in
`package.json`, so it is **ecosystem-neutral** (a Python/Rust/Go package carries the same file; no Node
coupling) and the core reads it directly. It is **the single source layer-1 harvests** (§0):
```jsonc
// <pkg>/.adhd/meta.json
{
  "area": "data", "group": "embed",        // MUST resolve to an area/group declared in .adhd/workspace.json (§8)
  "concerns": ["embedding", "provider-adapter"],
  "invariants": ["no area:platform import", "pure-TS"],
  "entrypoints": ["./dist/index.js"]
}
```
- **Name is DECOUPLED from folder path** (sox §2, adopted as a hard rule): the published name
  (`@adhd/<pkg>`) is the registry/content-address key — a rename/move must **never** change it (changing it
  breaks integrity/registry). The layout map (§8) governs *placement*; it never governs the name.
- `area`/`group` are validated against the repo taxonomy (§8) — an undeclared area/group fails lint. The
  generator can also *suggest* placement by matching a new package's `concerns` against the area/group
  **descriptions** (§8), so an agent places code from data instead of guessing.
- Required on every non-exempt package; the boundary policy (§9) and routing index (§0) both read it.

## 4. Required nx targets (every non-exempt project)  ·  Q5
`build` · `lint` · `test` (default-running; no env-gated live suites except CLAUDE.md's third-party rule)
· `typecheck` (catches *type* errors eslint's type-aware rules don't fully surface — **for non-TS projects
the `typecheck` target maps to the ecosystem checker, e.g. `mypy`/`pyright`; §11-res-4**) · `demo` (drives
`DEMO.md` against the real entrypoint) · `verify` (= lint + typecheck + test + demo; the pre-merge gate) ·
`nx-release-publish` for publishable libs (`dependsOn:["build","test"]`, `publishConfig.access:public`).
Publish *posture* is a generator flag (`--public`/`--private`, sox §2); the publish **action** stays
owner-gated regardless of posture.

## 5. Required files + sections (every non-exempt project)  ·  Q6
`README.md` · `CLAUDE.md` · `DEMO.md` · `CHANGELOG.md` · `PLAYBOOK.md`.
- **Existence = hard error.**
- **Still-placeholder / unchanged-template content = warn (dev) / error (production/release CI).**
- Required sections (warn/error as above): README → "Public API"; DEMO → runnable, driven by `demo`;
  CHANGELOG → an `Unreleased` entry; CLAUDE → "Invariants"; PLAYBOOK → pre/post-merge steps.
- The `CLAUDE.md` is also layer-2 (§0): group-level rules placeholder stamped at generation (sox §2).

## 6. Post-change enforcement layer  ·  Q6 (expansion) — **own feature, see §10**
A declarative "what must be updated when you change X" ruleset, checked as a gate (warn dev / error CI),
driven by a git diff + the metadata manifest (no nx):

| change detected | required update | severity |
|---|---|---|
| any `src/**` change | CHANGELOG `Unreleased` entry (with provenance, §7) | error (CI) |
| public-API change (exported symbol/signature) | README "Public API" + `DEMO.md` | error (CI) |
| new feature (new export/route/target) | `DEMO.md` + ≥1 default-running live test | error (CI) |
| dependency added/removed | manifest + CHANGELOG | error |
| breaking change | major version bump + CHANGELOG breaking note | error |

Generalizes CLAUDE.md's existing "disclose to BACKLOG / never bury bugs / changelog" rules into a checkable
layer. Reusable beyond workspace scaffolding → **broken out as `FEAT-CHANGE-ENFORCE-001`** (§10).

- **author** — `name:version` (AI: the agent spec's `version:` frontmatter; human: the git author)

## 8. Inheritance, upgrade, lint, layout  ·  Q1–Q4, Q8
**Inheritance (Q1) — automatic-first.** Configure the **root nx config** so the standard applies
automatically after *every* generate action; fall back to first-party generators calling
`applyWorkspaceStandard(tree, project)` as their final step **only if** nx can't trigger it for all
generators.

**Upgrade (Q2) — managed regions + tag confirmation.** `nx g @adhd/workspace-nx:upgrade [--project p]
[--dry-run]` idempotently re-applies the standard; managed files carry `@workspace:managed` marker regions so
hand-edits outside the markers survive. The upgrade/lint **confirms each managed file actually contains its
marker tags** — a file missing its markers is unmanaged = a violation. (Marker engine lives in the core, §1a.)

**Lint gate (Q3) — eslint workspace rule.** `@adhd/eslint-plugin-workspace` (nx adapter, §1b) gives in-IDE
feedback and fails CI when a project is missing a required target/file/field, a managed marker, or a
post-change-rule update; names the project + exact item + the `:upgrade` fix command.

**Scope (Q4).** All projects; explicit per-project `workspace: exempt` opt-out in the manifest.

**Configurable, repo-defined taxonomy + layout (Q8) — `.adhd/workspace.json`.** The taxonomy lives in a
**dedicated root file `.adhd/workspace.json`**, NOT `package.json` — because it is a first-class, repo-specific
artifact (and decouples from Node). **Each repo defines its own taxonomy**: adhd is *project-focused*
(entrypoints/workflows/components/ui/data/logic/shared), which is a different vocabulary from sox's
*db/ml/platform-centric* `area:{platform,data,shared}`. We adopt sox's **mechanism** (areas/groups +
depConstraint matrix + descriptions), never its vocabulary — adhd's primitives are adhd's own. The core treats
the whole taxonomy as data. `generate-lib.sh` is superseded by `@adhd/workspace-nx`.

**Each area and group carries a `description` (+ `whenToUse`/`examples`)** so an agent — and the generator's
placement-suggester (§3) and the layer-5 router (§0) — can determine *where something belongs from meaning*,
not by guessing at a bare label:
```jsonc
// .adhd/workspace.json (adhd's own, project-focused taxonomy)
{
  "scope": "@adhd",
  "areas": {
    "features": {
      "description": "Stateful user journeys + the merge point where UI meets data (GQL/DB).",
      "whenToUse": "A user-facing capability that wires components to data. Not a pure visual unit.",
      "groups": {
        "workflows":  { "description": "Multi-step stateful journeys + sub-routing.", "dir": "packages/features" },
        "components": { "description": "UI-meets-data unit; the merge point.",        "dir": "packages/features" }
      }
    },
    "design-system": {
      "description": "Pure visual units with no data dependency.",
      "groups": {
        "ui-primitives": { "description": "Atomic visual units (Button, hooks).",  "dir": "packages/design-system" },
        "ui-composites": { "description": "Complex pure-visual units (Cards).",     "dir": "packages/design-system" },
        "tokens":        { "description": "Design tokens.",                          "dir": "packages/design-system" }
      }
    },
    "shared": {
      "description": "Pure-TS foundation tools; safe in Node + browser; the leaf of the dependency graph.",
      "groups": {
        "shared": { "description": "Foundation tools (data, transform).", "dir": "packages/shared" },
        "logic":  { "description": "Domain business rules.",              "dir": "packages/shared" },
        "data":   { "description": "Data engines (query).",               "dir": "packages/shared" }
      }
    }
    // …entrypoints, testing, ai/mcp — each with descriptions + dir mapping
  },
  "platforms": { "node": "…", "browser": "…", "shared": "…" }   // adhd's platform-isolation axis (orthogonal)
}
```
The linter validates every package's `area`/`group` (§3) against this file; the routing index (§0 layer 1) is
generated from it + the metadata blocks.

## 9. Module-boundary enforcement + sox reconciliation  ·  (mechanism harvested from sox §3/§4/§7)
**Boundary policy as data (core, §1a), declared in `.adhd/workspace.json` (§8).** The depConstraint
allow-matrix is **repo-defined** over the repo's own areas/groups — adhd uses its *project-focused* downward
dependency flow (CLAUDE.md §1), not sox's db/ml vocabulary. adhd's matrix (illustrative):
- `area:features` → `features` + `design-system` + `shared` (+ entrypoints orchestrate downward only).
- `area:design-system` → `design-system` + `shared`.
- `area:shared` → **only** `area:shared` (pure leaf; the bedrock).
- plus the orthogonal `platform:*` isolation axis (node ↛ browser internals, browser ↛ node internals).
- (optional intra-area tightening, e.g. a `group:components` → `group:ui-primitives`, not the reverse.)

**Enforcer (nx adapter, §1b):** wired into `@nx/enforce-module-boundaries`. **Acceptance:** a synthetic
upward/cross-area import (e.g. `shared` → `features`) MUST fail lint.

**What we took from sox's handoff (mechanism only) — and the boundary going forward.** We harvested the
*ideas* once, from the handoff doc the user pointed us to. **adhd is now the reference implementation; we do
NOT pull from the sox repo going forward** (no shared vocabulary, no code copy). The core is published *from
adhd* (§11-res-5); sox's `sox-nx` can later become a second adapter over the adhd-published core.
- *Mechanism transferred to the generic core:* name-decoupled-from-path integrity rule (§3); the per-package
  metadata block schema (§3); the **area/group + depConstraint + description** mechanism *as data* (§8); the
  routing-index contract + drift gate (§0); the core↔adapter split (§1); the 4-point acceptance suite (below).
- *NOT taken (sox-specific, would be the wrong primitive for adhd):* sox's `area:{platform,data,shared}` +
  db/ml groups vocabulary (adhd defines its own, §8); the `registry/index.json` checksum mirror (BL-33);
  esbuild-self-contained build specifics.
- *adhd already has, beyond the sox handoff:* required-docs set (DEMO/PLAYBOOK/CHANGELOG), the post-change
  enforcement layer (§6), change provenance (§7), managed-region markers (§8) — these are adhd's lead.

**Acceptance suite (adopted from sox §7):**
1. `nx g @adhd/workspace-nx:library --area data --group vectors my-thing` → package at the layout-mapped
   path, name `@adhd/<…>` (path-decoupled), metadata block stamped, tags `area:data`+`group:vectors`,
   standard targets, public-ready manifest, all required files present.
2. registry drift gate green after generation (where a registry exists).
3. A hand-written cross-boundary import (`data/*` → a `platform/*` pkg) **fails** boundary lint.
4. The routing-index generator picks up the new package from its metadata with **zero hand-editing**.

## 10. BACKLOG decomposition  ·  (resolves residual #1 — split confirmed)
Given the core/adapter cut and cross-repo reuse, the big reusable subsystems are broken out so they aren't
confined to workspace scaffolding (or to nx):

| Feature | Scope | Depends on |
|---|---|---|
| `FEAT-WORKSPACE-001` | the standard + core (`@adhd/workspace-standard`) + nx adapter (`@adhd/workspace-nx`): targets, files, layout, inheritance, upgrade, boundary enforcement, routing index | the two below |
| `FEAT-CHANGE-ENFORCE-001` | post-change enforcement rule engine (§6) — git-diff-driven, nx-free | provenance |
| `FEAT-PROVENANCE-001` | change-provenance schema + validator (§7), tied to `/reflection` | — |

(Layer-1 routing index, layer-3 impact graph, layer-5 intent router from §0 are tracked under
`FEAT-WORKSPACE-001` as the standard's deliverables; layer-4 soft memory already exists via the memory MCP.)

## 11. Decisions — all resolved (2026-06-26)
1. **Split §6/§7 into own features → YES** (§10): `FEAT-CHANGE-ENFORCE-001` + `FEAT-PROVENANCE-001`.
2. **Provenance carrier → BOTH.** Git **commit trailers** (`Work-Item:`/`Author:`/`Model:`) are the
   enforceable source written at commit time; the **CHANGELOG** trailer is the human-visible projection
   generated from them. The post-change gate (§6) verifies both are present + well-formed.
3. **Author identity → resolved from the running agent's system-prompt (SP) context** — the same identity
   resolution `/reflection` uses: `SOX_AGENT_NAME` → `--- AGENT: <name> ---` header → the operating spec's
   `name:`/`version:` frontmatter. Human authors → git author. (No inference from loaded context.)
4. **`typecheck` for non-TS → MAP it** (mypy/pyright), don't exempt (§4) — a missing type gate is the exact
   blind spot the standard closes.
5. **Repo home → publish `@adhd/workspace-standard` from adhd.** adhd is the reference implementation; do not
   look at / depend on other repos. sox's `sox-nx` may later consume the adhd-published core as a second
   adapter.

### Config homes (resolved with the §8/§3 moves)
- **Root taxonomy + layout + boundary policy → `.adhd/workspace.json`** (repo-defined; areas/groups carry
  `description`/`whenToUse`/`examples`). Not `package.json`.
- **Per-package metadata → `<pkg>/.adhd/meta.json`** (ecosystem-neutral). Not `package.json`.
- The reframe's five layers are implemented as **generator + linter logic** (§0a), not docs alone.
