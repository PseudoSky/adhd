## [def:task_usage table]
```sql
CREATE TABLE task_usage (
  task_id       TEXT PRIMARY KEY,
  root_task_id  TEXT,              -- null = this task IS the root; non-null = ancestor root task_id
  agent_name    TEXT NOT NULL,
  provider_type TEXT NOT NULL,
  model         TEXT NOT NULL,
  input_tokens  INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL DEFAULT 0,
  tool_call_count INTEGER NOT NULL DEFAULT 0,
  model_calls   INTEGER NOT NULL DEFAULT 0,
  latency_ms    INTEGER NOT NULL DEFAULT 0,  -- 0 until terminal event; set on completed/failed/cancelled
  is_complete   INTEGER NOT NULL DEFAULT 0,  -- 0 = in-progress row; 1 = terminal event fired
  created_at    TEXT NOT NULL
);
```
Drizzle schema definition goes in `packages/ai/agent-mcp/src/db/schema.ts` as `taskUsageTable`. Migration generated by drizzle-kit.

`root_task_id` is resolved at the terminal event by walking `tasks.parentTaskId` up the chain until `parentTaskId IS NULL`. The root task itself has `root_task_id = NULL`. Sub-tasks store their root's `task_id`. This allows `WHERE root_task_id = ? OR task_id = ?` to retrieve all rows in a delegation tree without a recursive CTE.

`is_complete` distinguishes rows written mid-task (durability writes on each `post:model_response`) from rows that reached a terminal event. A row with `is_complete = 0` means the task is still running or the process crashed before the terminal hook fired.

## [inv:plugin-no-throw]
`UsagePlugin.install()` registers handlers that must never throw. All errors inside hook handlers are caught and logged via the pino logger. An observational plugin that crashes the host process is worse than no tracking at all.

## [inv:claudecli-undefined]
The claudecli provider does not return token counts. `ProviderChatResponse.usage` is `undefined` for claudecli tasks. `UsagePlugin` records this as a row with `input_tokens=0, output_tokens=0` — the row still exists so callers can detect the task ran; they should not interpret zeros as "zero tokens consumed".

## [inv:ephemeral-flushed]
Tasks created via `agent_name` mode (no session_id) are still written to `task_usage`. The orchestrator always fires `task:completed` / `task:failed` / `task:cancelled` hooks regardless of session mode (`orchestrator.ts:336`). So `UsagePlugin` receives the event and writes the row. The resulting `task_usage` row has no corresponding `tasks` row — that is expected and correct. Callers should not assume a `tasks` row exists for every `task_usage` row.

## [inv:incremental-write]
`UsagePlugin` writes to `task_usage` on every `post:model_response` event (UPSERT — insert on first call, update accumulators on subsequent calls). This ensures token data survives a process crash between the last model call and the terminal event. The terminal event (`task:completed` / `task:failed` / `task:cancelled`) does a final UPDATE to set `latency_ms`, `root_task_id` (from the in-memory accumulator — see `[inv:root-task-resolution]`), and `is_complete = 1`. A row with `is_complete = 0` in a live database indicates an in-progress task; in a crashed/restarted database it indicates partial data up to the last model call.

**Terminal hook await requirement:** `task:failed` and `task:cancelled` must use `await hooks.emit(...)` — not `void hooks.emit(...)` — to guarantee the terminal UPDATE commits before the orchestrator's `finally` block closes the registry. Using `void` risks leaving rows with `is_complete = 0` on failure/cancellation even when the task genuinely completed those paths.

## [inv:root-task-resolution]
`root_task_id` is resolved at **task-creation time** from the in-memory `callerContext` chain — not by walking the `tasks` DB table at terminal event. Ephemeral tasks never write a `tasks` row, so a DB walk would silently produce the wrong answer for ephemeral sub-tasks.

Derivation (in `taskTool` and `runEphemeralTask`):
```typescript
const rootTaskId = callerContext
  ? (callerContext.rootTaskId ?? callerContext.taskId)  // caller is root if rootTaskId is null
  : null;  // this task IS the root
```

`rootTaskId` is added to `ExecutionContext` and forwarded in `TaskStartPayload`. `UsagePlugin` reads it from the `task:start` payload and stores it in the in-memory accumulator. The terminal UPDATE writes it to the DB without any SELECT.

## [shape:UsageSummary]
```typescript
interface UsageSummary {
  inputTokens: number;
  outputTokens: number;
  modelCalls: number;
  toolCallCount: number;
  latencyMs: number;
}
```
Aggregated token counts for a single task or a subtree. Used inside `[shape:TaskUsageReport]`.

## [shape:TaskUsageReport]
```typescript
interface TaskUsageReport {
  direct: UsageSummary;   // this task's own model calls only
  subtree: UsageSummary;  // this task + all sub-tasks (root_task_id aggregation)
  taskCount: number;      // 1 if no sub-tasks, N otherwise
}
```
Returned in the `usage` field of `TaskToolOutput` and the enriched `resultTool()` response.

- `direct`: aggregates from `task_usage WHERE task_id = ?`
- `subtree`: aggregates from `task_usage WHERE task_id = ? OR root_task_id = ?`
- `subtree` equals `direct` when `taskCount === 1`.
- If no `task_usage` row exists for the task (task still running, or cancelled before first model call), `usage` is absent from the response.
- claudecli tasks: `usage` is present with all-zero counts — callers must not interpret zeros as "zero tokens consumed".
