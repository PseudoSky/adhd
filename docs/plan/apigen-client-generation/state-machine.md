## Parallel tracks summary

### Track A: Foundation (3-way parallel, barrier at audit-core)

- `schema-extraction`, `schema-composition`, and `nx-generator` all depend on `core-types` and can execute concurrently.
- `nx-generator` moved from CLI phase to foundation so the generator is proven before it is used.
- `audit-core` only gates `schema-extraction` + `schema-composition`; `nx-generator` must also be done before `scaffold-plugins` (which depends on both `nx-generator` AND `audit-runtime`).
- Barrier at `audit-core` before any runtime work begins.

### Track B: Runtime (parallel pair, barrier at audit-runtime)

- `runtime-middleware` and `runtime-dispatch` can execute concurrently.
- **MERGE PROTOCOL:** Both mutate `packages/apigen/runtime/src/index.ts`. Whichever completes second must merge all exports from both states into index.ts.
- Barrier at `audit-runtime` before any plugin work begins.

### Track C: Plugins (sequential scaffold → reference → checkpoint → 4-way parallel)

1. `scaffold-plugins` — sequential (runs generator ×5); depends on nx-generator + audit-runtime.
2. `plugin-api-fastify` — reference implementation; sequential after scaffold.
3. `plugin-fastify-checkpoint` — HUMAN GATE; requires manual approval file.
4. `plugin-jsonschema`, `plugin-mcp`, `plugin-api-express`, `plugin-cli-output` — 4-way parallel after checkpoint.
5. `audit-plugins` with reviewer gate before CLI work begins.

### Track D: CLI (2-way parallel, barrier at audit-cli)

- `cli-generate-cmd` and `cli-run-cmd` are parallel. (`nx-generator` is now in foundation — no longer here.)
- **MERGE PROTOCOL:** `cli-generate-cmd` creates `packages/apigen/cli/src/index.ts`. `cli-run-cmd` adds `run`/`run-registry` commands. The second state to complete merges index.ts.
- Barrier at `audit-cli` before integration tests.
